## What was Changed

### 1. Fixed `UnboundLocalError` in `journal_append`
The `journal_append` method previously relied on `os.open()` with a generic `finally: os.close(fd)` block which could throw an unbound error if the opening failed. 
- **Change**: Refactored to use standard python `with open(...)` which gracefully handles file opening/closing.

### 2. Fixed Race Condition in Parallel Tool Execution
File operations (`tool_write_file`, `tool_edit_file`, and `tool_replace_in_file`) executed via the `ThreadPoolExecutor` could silently step over one another when operating on the same file.
- **Change**: Imported `collections` and implemented a `FILE_LOCKS = collections.defaultdict(threading.Lock)` structure. Mutative file actions now acquire a per-path lock using `with FILE_LOCKS[full]:` to ensure atomic file access across the thread pool.

### 3. Fixed Context Window Blowout in `maybe_summarise`
The summary heuristic triggered exclusively on turn count (e.g., every 5 turns), leading to context blowout issues on dense token exchanges.
- **Change**: The function now iterates the `messages` array, checks the total accumulated payload size `total_chars = sum(len(str(m.get("content", ""))) for m in messages)`, and proactively triggers a summarization round if it breaches ~30k characters, keeping contexts lean and API-friendly.

### 4. Added Brittle Fallback Warnings for `tool_edit_file`
When exact diff matches fail, `tool_edit_file` defaults to whitespace-stripped search strategies (Fuzzy & Line-stripped modes). While robust for standard text, this behavior is destructive to whitespace-sensitive languages like Python or YAML.
- **Change**: If a non-exact match fallback strategy succeeds and the file extension ends with `.py`, `.yaml`, or `.yml`, the `tool_edit_file` observer now explicitly injects a prompt into context:
  > `WARNING: Fuzzy matching was used. For whitespace-sensitive files, this may have introduced indentation errors. Please verify the file structure.`
